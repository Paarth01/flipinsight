# FlipInsight — Flipkart Product Analytics Platform

[![CI](https://github.com/Paarth01/flipinsight/actions/workflows/ci.yml/badge.svg)](https://github.com/Paarth01/flipinsight/actions/workflows/ci.yml)

An end-to-end data analytics platform built on the [Flipkart e-commerce sample dataset](https://www.kaggle.com/datasets/PromptCloudHQ/flipkart-products) (20,000 product listings). Covers the full pipeline: **ETL → PostgreSQL → ML models → REST API → live dashboard**.

## Architecture

```
Raw CSV (20,000 rows, messy nested/stringified fields)
        │
        ▼
   etl/clean.py          — parses category trees, spec dicts, prices, ratings
        │
        ▼
   data/processed/products_clean.csv   (19,672 clean rows)
        │
        ▼
   db/init_db.py          — loads into SQLite via SQLAlchemy ORM
        │
         ├──► ml/train_price_model.py     — RandomForest price regressor (R² 0.77)
         ├──► ml/train_category_model.py  — TF-IDF + LogReg category classifier (98.2% acc)
         ├──► ml/train_recommender.py     — sentence-transformers (MiniLM-L6) semantic recommender
        │
        ▼
   api/main.py (FastAPI)  — REST endpoints for products, analytics, ML inference
        │
        ▼
   dashboard/static/index.html  — Chart.js dashboard + live prediction UI
```

## Key data quality fixes in ETL

The raw dataset has several messy fields that needed real cleaning, not just `pd.read_csv`:

- **Category tree parsing**: `product_category_tree` is a stringified single-element list like `["Clothing >> Women's Clothing >> Shorts"]`, split on `>>` into `category_l1..l4`.
- **Malformed category rows**: ~1.6% of rows have a broken tree with no `>>` separator, where the "category" is actually the full product title (e.g. `"Vishudh Printed Women's Straight Kurta"`). These were detected and excluded — without this fix, category cardinality was inflated from ~30 real categories to 265.
- **Specifications parsing**: `product_specifications` is stored in Ruby-hash syntax (`"key"=>"value"`), converted to proper Python dicts via `ast.literal_eval` after a `=>` → `:` substitution.
- **Ratings**: `"No rating available"` strings converted to proper nulls rather than being treated as text.
- **Discount calculation**: derived `discount_amount` and `discount_pct` from retail vs. discounted price, guarding against data-entry errors where discounted > retail.

## Running it

```bash
pip install -r requirements.txt

# 1. Clean the raw data
python3 etl/clean.py

# 2. Load into SQLite
python3 -m db.init_db

# 3. Train the ML models
python3 -m ml.train_price_model
python3 -m ml.train_category_model
python3 -m ml.train_recommender

# 4. Start the API + dashboard
python3 -m uvicorn api.main:app --reload
```

Then open **http://localhost:8000** for the dashboard, or **http://localhost:8000/docs** for interactive API docs (Swagger UI, auto-generated by FastAPI).

## Running with Docker

```bash
docker compose up --build
```

The image runs the ETL + ML training at **build time**. The database is seeded at **startup** via the lifespan hook — so it works with Railway, Render, or any managed PostgreSQL provider. Dashboard at **http://localhost:8000**.

To run with plain Docker instead of Compose:

```bash
docker build -t flipinsight .
docker run -p 8000:8000 flipinsight
```

### Switching to Postgres

The DB layer reads `DATABASE_URL` from the environment (defaults to local SQLite):

```bash
export DATABASE_URL="postgresql://flipinsight:flipinsight@localhost:5432/flipinsight"
pip install psycopg2-binary
python3 -m db.init_db
```

`docker-compose.yml` includes a PostgreSQL 16 service wired to the API via `DATABASE_URL`.

## CI

`.github/workflows/ci.yml` runs on every push/PR to `main`:
1. Installs dependencies
2. Runs the full pipeline from scratch (ETL → DB → train all 3 models)
3. Runs the test suite
4. Builds the Docker image and smoke-tests the container's `/health` endpoint

## Running tests

```bash
python3 -m pytest tests/ -v
```

19 tests covering health checks, analytics aggregation, product filtering/search, both ML prediction endpoints, and the recommender (similarity search + similar-by-ID).

## API endpoints

| Endpoint | Description |
|---|---|
| `GET /products` | Paginated product listing with filters (`category`, `brand`, `search`, `min_price`, `max_price`) |
| `GET /products/{uniq_id}` | Single product detail |
| `GET /analytics/summary` | Headline stats (totals, avg price, avg discount, rating coverage) |
| `GET /analytics/categories` | Product counts by top-level category |
| `GET /analytics/price-by-category` | Avg price & discount per category |
| `GET /analytics/top-brands` | Brand ranking by listing count |
| `GET /analytics/discount-distribution` | Discount % histogram buckets |
| `POST /predict/price` | Predict retail price from category/brand/description |
| `POST /predict/category` | Predict product category from description text (top-3 with confidence) |
| `GET /recommend/similar/{uniq_id}` | Find products similar to a given product |
| `POST /recommend/search` | Find products similar to a free-text query |

## Model performance

- **Price prediction** (RandomForest, log-target): MAE ≈ Rs. 1,056, R² ≈ 0.77 on held-out test set
- **Category classification** (TF-IDF + LogisticRegression, 25 classes): 98.2% test accuracy
- **Price prediction** (RandomForest, log-target): MAE ≈ Rs. 1,056, R² ≈ 0.77 on held-out test set
- **Category classification** (TF-IDF + LogisticRegression, 25 classes): 98.2% test accuracy
- **Recommender** (sentence-transformers MiniLM-L6-v2): dense semantic embeddings, cosine similarity search; free-text queries and product-ID lookups both supported
